今日のひとことまとめ

AIの出力が「良いか」を判断するには 正解（ground truth） が要る。でも正解は集めるのが大変でスケールしない。 だから 一部だけ正解を集めて、機械にパターンを学ばせる ── これが機械学習。 そのパターンの正体のひとつが 決定木（＝if文の自動化） で、 木は「2つの組がいちばんキレイに分かれる線」を 勘ではなく測定で 選んで育つ。

機械学習 自走学習プロジェクト ・ Day 1 ・ 2026-06-22
1タイトル

機械学習に入る前に、まず「普通の開発って何をしてるんだっけ？」を確認した。ここが全ての出発点。

具体例

アプリで条件分岐したいとき、こう書く：

if (残高 < 0) 警告を出す
if (年齢 >= 18) ...

しきい値も分岐も、ぜんぶ自分の頭の中の知識をコードに写しているだけ。

たとえ

Android / Flutter で毎日やってる、あの当たり前の作業そのもの。ルールを決める主体は「人間」。

気づき

普通の開発＝人間が手でルールを書く。これが効かなくなる瞬間から、機械学習の話が始まる。

手書きif文では歯が立たない問題がある。そこを機械にやらせる、というのが機械学習。

困りごと

「この写真は猫か犬か」を手書きif文で書ける？ ……書けない。「耳が尖ってたら猫」→ 垂れ耳の猫もいる。「ヒゲがあったら」→ 犬にもある。ルールが多すぎ・例外だらけ・そもそも言語化できない。

どう解決する

ルールを人が書く代わりに、正解つきの例をたくさん見せて、機械にルールを作らせる。

たとえ

「決定木の分岐は、要はif文のしきい値をデータから自動で決めているだけ」。後半でこれが効いてくる。

気づき

機械学習＝ルールの自動生成。人が書けない判定を、データのパターンから組み立てる。

「なぜAI案件でデータ分析・Kaggleなの？」への答えの中核。いちばん難しいのは "作ること" じゃない。

本当に難しいこと

AIに良い出力を「させる」ことより、その出力が本当に良いのかを判断して・測って・良くしていくことの方がずっと難しい。

自分ごとの例

Geminiチャットを「案件」として納品するなら、「良い感じ」では通用しない：

・出力の精度は何%？
・ハルシネーション（嘘）の頻度は？
・プロンプトA案とB案、本当にどっちが良くなった？

どれも勘では答えられない。集めて・基準を決めて・数えて・比べる＝データ分析が要る。

気づき

「動けば正解」のアプリ開発と決定的に違う。測れることが価値の土台になる。

「測る信号」を考えたら、すぐ落とし穴に当たった。これが評価のいちばん面白い所。

考えた信号

「1セッションのやり取り数」「まとめボタンを押したか」。直接測れない "深まり" を、観測できる行動で代用しようとした。これを 代理指標（proxy） と呼ぶ。

ワナ①：方向が定まらない

同じ「50往復」が、最高の深掘りとも、最悪のイライラ（噛み合わず言い直し／脱線）とも取れる。多い＝良い、とは言えない。

ワナ②：Goodhart

「往復数が多い＝良い」と決めて最適化すると、わざと往復が伸びる作りが "正解" になってしまう。指標を目標にした瞬間、その指標は壊れ始める。

気づき

やり取り数もボタンも、結局「外から行動を眺めて中身を推測してる」だけ。推測である限りズレは消えない。

外からの推測がダメなら、本当の答えはどこにある？ ── と問いを詰めて辿り着いた核心。

辿り着いた答え

「このセッションは思考が深まったか？」の本当の答えは、やり取りした本人の中にしかない。

用語

その「本人しか知らない本当の答え」を集めたものを 正解ラベル（ground truth） と呼ぶ。評価はいつも「本当の答えを握ってるのは誰／どこか」を突き止める所から始まる。

次のカベ

でも本人にしか分からない＝毎セッション・全ユーザーに聞いて回るのは無理（うざいし、スケールしない）。何百万セッションをどう判定する？

「全部は聞けない」を解くのが、まさに機械学習。最初の問いにここで繋がった。

アイデア

仮に 1000セッションぶんだけ 集める：
・各セッションの「外から見える信号」（往復数・まとめ押下・滞在時間）
・そのセッションに本人がくれた「本当の答え」（深まった／深まってない）

機械がすること

この山から「深まった人に共通する信号のパターン」を見つける。さらに本番では、まだ誰もラベルを付けてない新しいセッションにも判定を出せる ＝ 全部は聞けない問題が解ける。

用語

「特徴＋正解」から学んで新データに当てる枠組み＝教師あり学習。深まった／否の2択に振り分けるのが 分類。

「そのパターン、具体的にどんな形？」を手書きしてもらったら、決定木が出てきた。

自分で書いたコード

if (まとめ押した) { if (やり取り≥10) 深まった else 深まってない } else 深まってない

3つの手がかりで判定ロジックを手書きしたら、自然に入れ子のif文になった。

正体

それがそのまま 決定木。質問で枝分かれして最後に判定を出す。だから「決定木＝if文の自動化」。機械が学習した木と、形は完全に同じ。

気づき

しかも「滞在時間」は使わなかった＝無意識に "要らない手がかり" を捨てた。どれを使い／捨てるかも、決定木がやる仕事。

人間版と機械版の違いはたった1つ ── 「しきい値を勘で決めるか、データで決めるか」。その "データで決める" の中身。

出発点

「やり取り10回」は完全に勘だった。1000件の正解データを使って、最適な線引きを見つけるには？

重なり vs 分離
✗ 線が引けない
深まった ≒ 12回
深まってない ≒ 12回
→ 重なってる。どこに線を引いても両方混ざる。
✓ キレイに引ける
深まった = 14〜20回
深まってない = 2〜6回
→ 離れてる。間（約10）でスパッと分かれる。
原理

良い分割＝2クラスがいちばんキレイに分離する手がかりと線引き。機械は線を試し置きして "分離が最大" の所を選ぶ。どの手がかりを上に置くかも、同じく「いちばんキレイに分けるやつ」で決める。
→ 勘が、測定になった。

最後のピース。1回の分割ができれば、あとはそれを繰り返すだけで木が育つ。

しくみ（再帰）

① 全1000件で「いちばんキレイに分ける手がかり」を選ぶ（1階）。
② その枝に落ちたデータだけを取り出して、同じことをもう一回（2階）。
③ 枝ごとに繰り返す。

止めどき

ある枝の中身がほぼ一色（全部「深まった」など）になったら、そこで止めて判定を確定する＝葉。

たとえ

自分が手で書いた "2階建て" のif文を、機械はデータに従って何階でも自動で建てる。

一周つながった

正解を用意（燃料）→ 分離が最大の分割を測って選ぶ（エンジン）→ 枝ごとに再帰。これで決定木の学習が完成。
