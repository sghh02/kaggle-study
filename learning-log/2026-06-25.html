習得したキー概念

アンサンブル

ランダムフォレスト

多様な決定木を大量に作って多数決。1本の過学習を平均でならし、安定させる。

仕組み

バギング

木ごとに違う行で学ばせる（元データから復元抽出で引き直す）。木をバラけさせる1軸目。

仕組み

特徴量のランダム選択

各分割で使える列を一部だけに絞る。木をバラけさせる2軸目。

アンサンブル

GBDT（勾配ブースティング）

木を逐次に建て、前の木が間違えた所を次の木が直す。森より強いが過学習しやすい。

ツマミ

learning_rate（学習率）

各修正の "歩幅"。小さいほど慎重で過学習しにくいが、必要な木の本数が増える。

評価

交差検証（cv）

データを等分し、テスト役を交代させて全部で採点し平均する。1回の抜き打ちより公平な物差し。

注意

feature_importances

各列が稼いだ純度(gini)の合計。分割点が多い連続値が過大評価される "罠" がある。

手法

特徴量エンジニアリング

人間が知識で、モデルに無い情報を列にして渡す。効くのは "情報のギャップ" がある時だけ。

現象

過学習（再訪）

train は上がるが test は置いてけぼり。深すぎる木でも、木の足しすぎでも起きる。

結論：同じ "木 vs 森 vs GBDT" でも、勝者はデータの複雑さで入れ替わる。そして特徴量は、モデルが既に持っている情報の組み替えでは 効かない。

途中で踏んだ "ワナ"（実務の勘所）

森とGBDTは性格が逆：ランダムフォレストは木を増やしても悪化しにくいが、GBDTは 増やしすぎると過学習（test が山なりに下がる）。"とりあえず増やす" が通用しない。
比較は測り方を揃える：cross_val_score は cv を省くとデフォルト5分割。森を cv=10、GBDTを cv=5 で比べるのは "長さの違う物差しで身長比べ"。
feature_importances の連続値びいき：fare/age は分割点が無数にあり重要度を稼ぐが、二択の sex は過小評価される。"一番キレイに分ける" 列とズレる。鵜呑み禁物。
特徴量は万能じゃない：持っている列の組み替え（足し算・しきい値）は新情報ゼロで効かない。効くのは "情報のギャップ" がある時だけ。作ったら 必ず cv で測って見極める。

1タイトル

Day 3 で「木1本は過学習する」とわかった。その弱点をどう乗り越えるかが出発点。

困りごと

深い木は丸暗記（train98% / test73%）、浅い木は安定だが頭打ち。木1本＝1人の判断は、偶然のクセに引っ張られて偏る。

たとえ

開発で1人の実装をそのままマージせずレビューを入れるように、1人の判断に全部を委ねない。大勢に独立に見積もらせて平均すると、各自の偏りが打ち消し合う（集合知・ダーツの平均が中心に寄る）。

気づき

間違いの "方向" が木ごとにバラバラなら、平均で打ち消される。一方、本物のパターンはどの木も共通で当てるので平均しても残る。

「平均すれば良い」には条件がある。同じデータに同じ木を作ったら全部同じ＝平均する意味がない。

具体例

バギング＝木ごとに違う行で学ぶ（元の400件から復元抽出で400件引き直す＝重複や欠けが出る）。特徴量のランダム選択＝各分割で使える列を一部だけに絞る。

気づき

この2つで木をわざとバラけさせ、多数決する。これがランダムフォレストの "Random" の正体。決定木を対話だけで森まで拡張できた。

「森は木1本より強いはず」と自信満々で測ったら、予想が外れた。

困りごと

合成データを 10分割の交差検証で測ると、木1本(深さ3) 0.797 ＞ ランダムフォレスト 0.768。森が負けた。

気づき

森が弱いのではなく、単純なデータには単純なモデルが最適。きれいな分かれ目を浅い木が一発で掴めるので、森の "ブレ取り" の出番がなかっただけ。

森が本領を出せる土俵＝「木1本では掴みきれないデータ」を用意して名誉回復。

具体例

交互作用（cue1 XOR cue2 等）＋ 囮のノイズ列を仕込んだ複雑データでは、ランダムフォレスト 0.795 ≫ 木1本 0.681。深い木1本はむしろ 0.634 に悪化。

気づき

モデルの複雑さは、データの複雑さに合わせる。森が圧勝するのは、一本の木では捉えきれない複雑なデータのとき＝現実／Kaggle。

森は "独立に多数決" 止まり。誰も前の人の失敗を踏まえていない。そこを変える。

たとえ

ラフに書く → レビューで指摘 → そこだけ直す → また指摘…の繰り返し。次の木の仕事は、前までの木が間違えた所を狙い撃ちで直すこと（＝最初に自分で言った "改善を繰り返す" 道B）。

困りごと

直し続けると、直しようのないノイズまで暗記し始める。実際 test は 75本でピーク → 以降ジリ下がり、train は 1.000＝完全な丸暗記。

気づき

Day 3 の "深さカーブ" と同じ山なりが、今度は "木の本数" の軸で出る。森は増やしても悪化しにくいが、GBDTは 増やしすぎ注意。

暴走を止めるツマミ＝learning_rate（各修正の歩幅）。

具体例

歩幅を小さく（0.05〜0.1）＋木を多めにすると、増やしすぎても崩れにくい＝安全マージンが広い。代償は計算量が増えること。

たとえ

レビューで "一度に全書き換え"（大股）より、"小さく直して様子見"（小股）の方が事故らない。

気づき

比較は両者を同じ測り方で。cross_val_score は cv 省略でデフォルト5分割になり、森(cv=10)と揃わない＝フェアじゃない。"長さの違う物差しで身長比べ"。

道具立てが揃ったので、本筋を自分の手で書いた。

具体例

GradientBoostingClassifier(learning_rate=0.1, n_estimators=300) を自作。複雑データで 森 0.795 を 0.806 で上回った。歩幅小×本数多のバランスが効いた（lr0.05×400本だと "練習不足" で負ける）。

気づき

差は小さい（ほぼ誤差）が、「道B（逐次改善）は、ちゃんと調整すれば最強の森に勝てる」を実証。これが Kaggle で GBDT / LightGBM が定番武器な理由。

ここまで全部 "モデル交換"。今度はデータ側を人間が工夫する＝特徴量エンジニアリング。題材は本物の Titanic。

困りごと

仮説（女性・子供・身分が効く）は当たっていたのに、family_size / is_child / title を自作しても、全部ベースライン 0.776 を超えられない（誤差範囲）。

具体例

title（Mr/Mrs/Miss/Master）は生存率が 0.16〜0.79 で激しく割れる "優秀な情報"。なのに効かない＝中身は sex＋age で、GBDTが既に持っているから。

気づき

良い特徴量＝モデルが自力で取り出せない "新情報" を足すもの。偽データのXORは隠れて掘れない→FE劇的。Titanicは生の列に信号があり掘れる→FE効かず。FEが効くのは情報のギャップがある時だけ。だから必ず測って見極める。
