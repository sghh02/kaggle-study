習得したキー概念

道具

scikit-learn

Python の機械学習ライブラリ。決定木などのモデルが「完成品」として入っている。Day 1 で設計したエンジンの実装。

共通の作法

fit / predict

sklearn のモデル共通の3段階：①箱を作る → ②fit で学習 → ③predict で使う。1個覚えれば全モデルで使い回せる。

Python の作法

キーワード引数

ノブが多いので「どの設定か名前で指定」を強制。第1ノブ criterion の既定値が "gini"。Kotlin の名前付き引数の必須版。

評価の核心

精度 vs 純度

精度＝当たりの数（予測してから分かる）。純度＝クラスの片寄り（予測する前から付いている）。木は純度で分割を選ぶ。

純度の物差し

gini（不純度）

グループの混ざり具合。0＝一色（純粋）、最大 0.5＝半々（最も濁る）。低いほどキレイ。各クラスの生の件数から計算する。

モデルの中核

fit の自動化

全特徴量 × 全しきい値を試し、gini が一番下がる分割を選び、枝ごとに再帰。Day 1 で設計した処理を 1 行で実行する。

題材にした通し例

「思考を深めるチャットアプリ」を判定する決定木（Day 1 から継続）

「このセッションは思考が深まったか？」を、400 件の合成セッションデータ（chat_sessions.csv）で判定。隠れルールは Claude しか知らない ── 機械がそれを再発見できるかが検証。

機械に渡した手がかり（X）：
turns（やり取り数） used_summary（まとめ押下） stay_minutes（滞在時間＝囮）
本当の答え（y）：
deepened（深まった＝正解ラベル）

機械が組んだ決定木（max_depth=3 の一部）

turns ≤ 16.5 ?
YES（≤16.5）↓
used_summary ≤ 0.5 ?
YES（まとめ押してない）↓ ── ここがパズルの核心
turns ≤ 9.5 ?
YES ↓
深まってない（否53 / 深9・gini 0.248）
NO ↓
深まってない（否29 / 深14・gini 0.439）
NO（>16.5）↓
深まった（多数）

パズルの答え：分けても左右どちらも「深まってない」＝精度は 82/82 で不変。でも純度（gini）は親 0.342 → 子の合算 約0.33 に低下した。この「純度の改善」こそ機械の"得"。そして勘で一番上に置いた used_summary は2段目に落ち、囮の stay_minutes は重要度 0.001 でほぼ未使用 ── 一番効いたのは turns（重要度 0.764）だった。

途中で踏んだ "ワナ"（実務の勘所）

位置引数の罠：DecisionTreeClassifier(8) の 8 は、本当は max_depth ではなく第1ノブ criterion に向かう。ノブが多いライブラリは max_depth=3 のように名前付きで渡す。sklearn はそれを必須化して事故を防いでいる。
勘 vs 測定：「used_summary が一番効く」は自然な勘。でも機械は測って turns を選んだ。範囲が広い（1〜29）特徴は切り分け力が大きい。勘は出発点、結論はデータで確かめる。
精度の罠：「良い分割＝当たりが増える」と思いがち。でも当たりが増えなくても純度が上がる分割を、精度は見落とす。だから決定木は精度ではなく gini を基準にする。

決定木は「どこで分けるか」を、精度（当たりの数）ではなく 純度（gini） で決める。各クラスの生の件数を見て、混ざりが一番減る線を選び、枝ごとに再帰する ── その全部を、自分が書いた fit の1行 が自動でやっていた。

次回の入口

  • 1gini の式 ── 「当てずっぽうが外れる確率」が、なぜ 1 −（各割合の二乗の和） になるのか。記号を今日の直感に対応づける（Day 1 からの宿題の仕上げ）。

Android で「この UI はこう動くはず」と頭で完璧に設計しても、エミュレータで走らせて初めて腹に落ちる。今日はその "実機確認"。

「頭で分かる」と「手が動く」の間には毎回ギャップがある。Day 1 で 100% 思考だった分、今日はそこを埋めにいく。

道具を入れ、Day 1 の自分の例をデータにした（どちらも "足場"）。

具体例

uv で隔離環境を作り scikit-learn / pandas / matplotlib を導入。Day 1 の「チャットが深まったか」を 400 件の合成データ chat_sessions.csv にした（列：turns / used_summary / stay_minutes / deepened）。

たとえ

Flutter の pub get でプロジェクトごとに依存を閉じ込めるのと同じ発想。グローバルを汚さない。

気づき

隠れルールは Claude しか知らない。機械がそれ（＝自分が Day 1 で勘で描いた木）を再発見できるか、が今日の検証になる。

教師あり学習の基本作法を、自分の手で書いた。

具体例

X = df[["used_summary","turns","stay_minutes"]] で手がかり、y = df["deepened"] で正解。最後に clf.fit(X, y) で学習。

たとえ

ユニットテストの (入力, 期待値) ペアを 400 個渡して、その関数（木）を機械に書かせるイメージ。普通の開発の逆向き。

気づき

正解 y を X に混ぜてはいけない。混ぜたら「答えを見て答える」カンニング機になる。ground truth は "見せない答え" として持つ。

clf の箱を作る所でエラー。これは叱られたのではなく、作法を教わった瞬間。

困りごと

DecisionTreeClassifier(8) → エラー。8 を名前なし（位置引数）で渡したのが原因。

具体例

sklearn の第1ノブは criterion。8 はそこへ行こうとして弾かれた。正しくは max_depth=3 と名前付きで渡す。

たとえ

Kotlin の setMargin(top = 8) ── 名前付き引数。決定木はノブが十数個あるので、sklearn はそれを必須にしている。

気づき

ライブラリが意地悪なのではなく、事故防止。「どのノブを回すのか名前で言え」ということ。

実行して、自分の予想と機械が出した木を突き合わせた。

具体例

予想 = 一番上は used_summary / turns の線は 3 / stay_minutes は使う。機械 = 一番上は turns（≤16.5）。重要度は turns 0.764・used_summary 0.236・stay_minutes 0.001。

気づき

勘 vs 測定が具体化した。used_summary は 0/1 の二択、turns は 1〜29 と幅広く動くので切り分け力が大きかった。一方 stay_minutes はほぼ捨てられた ── Day 1 で「囮を無意識に捨てた勘」は当たっていた。

木の中に「分けても結論が同じ」という不思議な分岐を見つけた。

困りごと

turns ≤ 16.5 かつ used_summary ≤ 0.5 の中を、さらに turns ≤ 9.5 で分割。でも左（否53 / 深9）も右（否29 / 深14）も判定は「深まってない」。なぜ分けた？

気づき

結論（多数決）が変わらないのに機械が手間をかけた＝何か "得" があるはず。max_depth の許可があっても、得が無ければ機械は分けない。

精度を実際に数えてみたら、謎が解けた。

具体例

分けない時 = 当たり 82 / 外れ 23。分けた後も左右の合計で当たり 82 / 外れ 23。精度はゼロしか改善していない。

気づき

精度が変わらないのに分けた＝決定木は精度を基準にしていない。最大化しているのは「純度（各グループの片寄り）」だと自力で到達した。

純度の物差し gini と、fit が裏で自動でやっていたことを確認した。

具体例

gini ＝ 混ざり具合。0 ＝ 一色、最大 0.5 ＝ 半々。あの分割は親 gini 0.342 → 子の合算 約0.33 に低下。これが「得」の正体。

たとえ

袋からボールを1個取り、袋の割合通りの当てずっぽうで言い当てて外れる確率＝ gini。一色なら必ず当たる（0）、半々なら半分外す（0.5）。

気づき

fit は「全特徴量 × 全しきい値を試し、gini が一番下がる分割を選び、枝ごとに再帰」を 1 行で自動化していた。gini を使うのは criterion の既定が "gini" だから。純度は予測前のクラス件数で測る＝予測後に分かる精度とは、材料もタイミングも違う。
