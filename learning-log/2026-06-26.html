今日のひとことまとめ

LightGBMを「みんな使うから」ではなく なぜ sklearn の GBDT では足りないか から組み上げた。核心は3つ ── histogram（連続値を255箱に丸めて分割候補を固定→実測で約85倍速い）、leaf-wise（一番得する葉だけ深掘りする育て方）、early stopping（validationを見張り木の本数を自動決定）。最大の気づきは「生のLightGBMは精度の魔法ではない（同条件ならGBDTと同じ土俵）」。強さの正体は 速さ → 試行回数 → そこで精度を稼ぐ。最後に「表データの予測ならまずLightGBM」と実務の使いどころまで腹落ちした。

習得したキー概念

速さ

histogram（ヒストグラムベース）

連続値を255箱(bin)に丸め、分割候補を件数に依らず一定にして高速化。実測で約85倍。

ツマミ

max_bin

箱の数。デフォルト255。細かい差が効くデータなら増やせるが、ほぼ困らない。

育て方

leaf-wise

一番lossが下がる葉だけを伸ばす best-first 成長。level-wise（＝幅優先/BFS）の対。

ツマミ

num_leaves

leaf-wiseの過学習を抑える主役。葉の総数の上限（max_depthの代わり）。

自動化

early stopping

validationを見張り、改善が止まったら木の追加を自動で止める。本数を当てに行かない。

ツマミ

stopping_rounds

「何回連続で伸びなきゃ諦めるか」＝我慢の回数。ケチると早すぎ停止＝学習不足。

評価設計

3分割（train / valid / test）

学習・止め時判断・最終採点を分ける。validで止め時、testは触らず最後だけ。

ワナ

リーク（leakage）

止め時の判断にtestを覗くと、その特定データに"たまたまハマる"点を選び、test点が盛られる。

パラメータ

learning_rate × n_estimators

歩幅 × 歩数。小さい歩幅(lr)は多い歩数(本数)が要る。セットで考える。

途中で踏んだ "ワナ"（実務の勘所）

learning_rate は歩幅：42にしたら 0.475（コイン投げ＝発散）。lr=小さな歩幅(ブレーキ)、n_estimators=歩数。小さい歩幅には多い歩数が要る、をセットで持つ。
木の本数は手で当てない：n=10は過少(0.704)、多すぎは丸暗記。"ちょうど良い本数"はデータ次第 → early stopping に任せる。
リークの正体：止め時にtestを覗くと、その特定データに"たまたまハマる"点を選んでしまい、test点が"盛られた数字"になる。だから train/valid/test の3分割。
stopping_rounds はケチるな：5は凪に騙されて早すぎ停止＝学習不足(0.708)。十分(50)なら本当の頂点(106本/0.760)で安定。唯一の危険はケチること。
精度のタダ飯は無い：生のLightGBMは同条件でGBDTと同じ土俵（むしろ小データではやや下）。強さは速さ→試行回数で稼ぐもの。

1タイトル

Day5までは sklearn の GradientBoosting。本物のKaggleデータは2000件ではなく50万〜100万件。

困りごと

決定木は最良の分割を探すとき、列のしきい値を片っ端から試す（Day2）。候補数は最大で件数ぶん（50万行なら50万通り）、しかも1候補ごとに全行をなめる → 大データで激遅。

気づき

重いのは「木の本数(n_estimators)」ではない（本数は自分で決めるパラメータ）。重いのは 1分割の探索コストが件数で効いてくる こと。予想Bは方向だけ正解、原因は研ぎ直した。

「候補が多すぎ」をどう減らすか、を自分で考えた。

最初の案と、その破綻

二分探索（真ん中→さらに半分→…）を考えた。が、giniが下がる "谷" は複数できる（非単調）ので、真ん中を見ても best が左右どちらか安全に言えない＝枝刈りできず破綻。

たとえ

タイムスタンプをミリ秒で全部区別せず日付に丸める／画像を1pxずつ見ずブロックに粗くまとめる ── と同じ。木を建てる前に各列を255箱(bin)に丸め、候補を常に255に固定。

気づき

「適応的に絞る」のをやめ「前処理で一括ビニング」にした瞬間、枝刈りの悩みごと（どの半分を捨てるか）自体が消えた。255個なら全部試せばいい。

口で説明せず、手持ちデータで sklearn GBDT と直接対決して測った。

実測

10万行で sklearn GBDT 94.96秒 vs LightGBM 1.11秒 ＝ 約85倍。精度(cv=10)は GBDT 0.786 vs LightGBM 0.776 でほぼ互角（予想Bが的中）。

気づき

丸めて捨てた解像度には予測に使える情報がほぼ無い。木はYES/NOの粗いカットしかしない ── 23.41歳と23.43歳で切っても同じ人が左右に分かれるだけ。だから255箱で意味ある境目は全部拾える。

精度は上がらないのに、なぜ現場で「最強」と呼ばれるのか？という矛盾を解いた。

矛盾

生で回すとGBDTに勝てない。なのに「LightGBM最強」は本当。外していたのは "なぜ強いか" の部分だけ。

気づき（自分で到達）

速さは "精度そのもの" ではなく 検証回数（ハイパラ・CV・特徴量実験） に化け、その中から良い設定を引いて精度を稼ぐ。おまけに sklearn では現実的に扱えない巨大データも回せる＝そもそも土俵に立てる。

ここは本筋なので自分で記述。予想は強気で「めっちゃ上がる」。

2つの事故から学んだ
lr=42（事故）
0.475 ＝ コイン投げ＝発散。ブレーキを42回踏み込んだ状態。
n=10（過少）
0.704。歩幅0.1で10歩しか歩かずゴール未到達。
n=300, lr=0.1（同条件）
0.775。これで初めてDay5と公平比較。
たとえ

learning_rate＝歩幅（1本でどれだけ進むか／ブレーキ）、n_estimators＝歩数。小さい歩幅には多い歩数が要る。

気づき

結果は GBDT 0.806 / RF 0.795 より下。予想は外れ。生のLightGBMは精度の魔法ではない＝この外れが「強さの正体は速さ」への入口になった。

「葉の使い方が違う」と「255bin」を混同しかけたので、別物だと切り分けた。

たとえ（探索で）

level-wise(sklearn GBDT)＝幅優先探索(BFS)：階ごとに揃えて広げる対称な木。leaf-wise(LightGBM)＝最良優先(best-first)：優先度付きキューで「一番lossが下がる葉」を1個ずつ伸ばす、いびつな木。

過学習カーブ

trainは201本で1.000（丸暗記）、testは150本前後で頭打ち（0.78）。test "ピーク"581本は伸びでなく平らなさざ波の最高点。

気づき

leaf-wiseの高い表現力は諸刃 ── 賢く育つが暗記も速い。だから歯止めは max_depth ではなく num_leaves（葉の総数上限）。「伸び代」は test ではなく train(暗記)に注がれていた。

手で本数を当てる作業を機械に渡す。発想の転換：n_estimatorsは "天井"。

コード（自分で書かず解説で受け取った）

n_estimators=2000(天井)、eval_set=[(X_val,y_val)](見張る模試)、early_stopping(stopping_rounds=50)(50回伸びなきゃ停止) → best_iteration_=106、触ってないtestで0.760。

なぜtrainで判断しちゃダメ

trainはすぐ1.0で天井＝止め時を教えない。だから初見データ(validation)を見張る。

気づき（リーク）

止め時の判断に本物のtestを覗くと、その特定データに"たまたまハマる"点を選び、test点が盛られて物差しが嘘をつく＝リーク。だから train/valid/test の3分割：validで止め時、testは最後の正直な採点だけ。

stopping_rounds（我慢の回数）を変えて挙動を実測。

実測
rounds=5
22本で停止 / test 0.708 ＝早すぎ＝学習不足
rounds=20 / 50 / 200
すべて106本 / test 0.760 ＝十分なら安定
気づき

カーブはギザギザ（ノイズ）。我慢が足りないと一時の "凪" を頭打ちと勘違いして取りこぼす。一度十分にすれば結果は安定 → stopping_rounds はケチらず多め（50が定番）。唯一の危険はケチること。

「そもそも実務でいつ使うの？」を最後に接続。これで道具に血が通った。

使う場面

表(テーブル)データで、ある列を他の列から予測したいとき：離脱予測／課金予測／不正検知／需要予測／レコメンドのスコアリング。実務データの大半がこの "表" の形。

たとえ（自分の世界で）

ずっと題材にした「チャットが深まったか(deepened)判定」が、まさに分類タスクのミニチュア実務。アプリのバックエンドに predict を置けば、ユーザーデータ→スコアが返る関数になる＝アプリ開発と地続き。

気づき

「AI＝ディープラーニング」ではない。画像/音声/テキストはディープ、表データならまずLightGBM（Kaggleのテーブルコンペも実務も初手はこれ）。今日その反射の中身を理解した。
