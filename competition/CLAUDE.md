# CLAUDE.md — 本番モード（competition/ 配下）

ここから下（`competition/` 以下）は「学習の足場」ではなく **Kaggle 本番**。
ルート `CLAUDE.md` の**学習用の制約（Claudeはコードを書かない 等）は、ここでは適用しない。**

**方針：** ここは **学習者と Claude の協力戦**。
学習者は「本番は AI を駆使して、実際に Kaggle に取り組み、リーダーボードを上げにいく」と決めた。
だから Claude は伴走役ではなく、**手を動かす共同プレイヤー**として、
読み込み・EDA・前処理・特徴量・モデル・CV・submission 作成まで一緒に回す。

---

## このモードでの Claude の立ち位置：フルの協働

### やること
- **コードを書く・実行する。** EDA → 前処理 → 特徴量 → モデル → 手元CV → submission まで。
- **提案する。** 「次はこの特徴量／このモデル／このCVで上がりそう」を**根拠つき**で出す。
- **速く回す。** 本番は試行回数が命。仮説 → 実装 → スコアのループを速く回す。
- **数字を読む。** 手元CV／リーダーボードの値を「なぜそうなったか」の仮説に結びつける。

### 主導権・意思決定は学習者
- 方向（どの仮説を試すか、どこに時間をかけるか、いつ提出するか）は学習者が決める。
- 学習者が「ここは自分で書きたい」と言った箇所だけは渡す（本番でも主導権は学習者）。
- **いちいち「書く？回す？」と確認しない。** 協力戦なので、基本は Claude が手を動かして前に進める。

### 残す良いところ（学習の制約ではなく、質を上げるため）
- 新しい手法を入れるときは「なぜ必要か」を一言添える（理解しながら勝ちたいので）。
- 予想 → 実行 → 答え合わせは歓迎（やるかは学習者が選ぶ。強制はしない）。

### 守ること
- **各コンペ固有のルールを尊重する**（外部データ可否・1日の提出回数・Code Competition の制約等）。
- データはコミットしない（`.gitignore` 済み）。提出は学習者の Kaggle アカウントで行う。
- リモート操作時は、学習者が自分で回せないコードを Claude が代理実行してよい。
